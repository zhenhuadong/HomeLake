工作区文件有四种状态
untracked   : 没track
unmodified  ：已经track, 还没modify, 就是没改动
modified    ：已经track, 现在modify了，还没有提交本次改动到版本库
staged      ：已经track, 已经提交到版本库了

如何反悔：
================================================================================
Q1：我想丢弃当前的修改怎么办？
A1：分五种情况，
1. 将工作区内新建的文件丢弃 (处于untracked状态的文件)
"git clean -n ." (查看要丢弃的新建文件)
"git clean -f ." (丢弃全部新建的文件)
"git clean -f filename" (丢弃新建的文件)

2. 将工作区内的改动丢弃  (处于modified状态的文件)
"git checkout ."             在工作区内的改动全部丢弃
"git checkout filename"      在工作区内的该文件的改动丢弃

3. 将暂存区内的改动丢弃  (处于staged状态的文件)
"git reset ."           全部未commit的index内容打回工作区   (然后参考2)
"git reset filename"    该文件未commit的index内容打回工作区 (然后参考2)

"git reset --soft"           全部未commit的index内容打回工作区   (然后参考2)
"git reset --hard"           全部未commit的index内容全部丢掉

4. 将已提交到local repository中commit丢弃
"git reset --soft HEAD^" 将最后一次commit以及之后的修改全部打回工作区
"git reset --hard HEAD^" 完全丢弃最后一次commit以及之后的修改
"git reset --soft commit" 将该commit之后的修改全部打回工作区

5. 将已提交到central repository中的commit丢弃

如何处理冲突：
================================================================================
Q2:我如何处理遇到的冲突？
A2:分清楚是什么类型的冲突：逻辑冲突, 树冲突, 内容冲突
1. 逻辑冲突
git自动处理（合并/应用补丁）成功，但是逻辑上是有问题的。
example: User1 move file A to file B, while user B has file C still use file A. git merge can be successful, but it is not correct now.
example: function A return int,  sb change the reture value to bool, it is ok to merge. but the function invoked this procedure is not correct any more
GIT不去解决这些冲突,需要人去解决和控制，很多时候就是一个bug.

2. 树冲突
文件名修改造成的冲突，称为树冲突。
比如，a用户把文件改名为a.c，b用户把同一个文件改名为b.c，那么b将这两个commit合并时，会产生冲突。
git status
    added by us:    b.c
    both deleted:   origin-name.c
    added by them:  a.c
如果最终确定用b.c，那么解决办法如下：
git rm a.c
git rm origin-name.c
git add b.c
git commit

3. 内容冲突
两个用户修改了同一个文件的同一块区域，git会报告内容冲突。内容冲突很常见，一般发生在merge/patch, 或者 rebase中。

3.1. merge/patch的冲突：需要commit
先编辑冲突，然后git commit提交。
对于git来讲，编辑冲突跟平时的修改代码没什么差异。修改完成后，都是要把修改添加到缓存，然后commit。
<<<<<<< HEAD
b789               <--- 我的修改
=======
b45678910          <--- 别人的修改
>>>>>>> 6853e5ff961e684d3a6c02d4d06183b5ff330dcc

可以使用可视化的tool修改， 配置可用下面的命令
git config --global merge.tool gvimdiff
merge tool candidates: opendiff kdiff3 tkdiff xxdiff meld tortoisemerge gvimdiff diffuse ecmerge p4merge araxis bc3 codecompare emerge vimdiff
usage:
git mergetool


3.2 rebase的冲突:  不需要commit
rebase的冲突解决过程，就是解决每个应用补丁冲突的过程。
解决完一个补丁应用的冲突后，执行下面命令标记冲突已解决（也就是把修改内容加入缓存）：
git add -u
然后继续
git rebase --continue

如果中间遇到某个补丁不需要应用，可以用下面命令忽略：
git rebase --skip

如果想回到rebase执行之前的状态，可以执行
git rebase --abort
